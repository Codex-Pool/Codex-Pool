# 核心管理页重设计需求稿（交付设计师）

## 1. 文档目的
本需求稿用于指导独立设计师为 Codex Pool 重做 4 个核心管理页原型：

1. 管理员登录页
2. Dashboard（服务总览）
3. 账号池
4. 批量导入

这次任务是“重新设计”，不是“在现有界面上做美化微调”。

设计师可以彻底推翻现有视觉表现与布局结构，但必须保留业务目标、核心流程、页面状态、关键动作与信息约束。

## 2. 项目背景
当前前端虽然已经做过一轮收敛，但整体仍不满足以下要求：

1. 仍然带有明显后台模板气质，缺少真正成熟、耐用、长期可用的专业控制台质感。
2. 页面组织仍偏“卡片岛”，没有形成强主次、强秩序、强工作面的连续体验。
3. Dashboard、账号池、导入页都还更像“组件拼装页”，而不是被编辑过的专业软件界面。
4. 登录页也仍然没有达到“第一眼可信、清楚、克制、专业”的入口标准。

本轮交付给设计师的目标，不是修饰，而是建立一套可以作为后续前端总方向的原型基线。

## 3. 产品定位与用户
### 3.1 产品定位
Codex Pool 是高频使用的 B2B 控制台产品，不是营销站点，不是极客实验工具，也不是开发者自嗨的可视化面板。

它服务的是长期运维、排障、导入、授权、计费核对、路由调整等连续工作任务。

### 3.2 核心用户
1. 平台管理员：高频进入后台，关注稳定性、可控性、风险、效率和追踪能力。
2. 自托管部署者：通常既是管理员也是维护者，关注系统状态、配置一致性和可排障性。
3. 小团队运营者：需要快速定位问题、导入账号、检查用量、管理密钥与上游资源。

### 3.3 用户心智
1. 登录页：用户希望“尽快、安全、明确地进入工作台”，不希望看到模板化欢迎页。
2. Dashboard：用户希望“先快速判断哪里有问题，再决定点进去看哪里”。
3. 账号池：用户希望“高密度地看清账号状态，并立即执行批量管理”。
4. 批量导入：用户希望“清楚知道导入前是否安全、导入后进度如何、失败在哪一条”。

## 4. 设计方向与全局约束
### 4.1 设计北极星
本轮设计必须遵守以下方向：

1. 高端精致、可信克制、专业清晰。
2. 像“被认真编排过的专业桌面控制台”，而不是“被 AI 精修过的后台模板”。
3. 以连续工作面、信息秩序、控件压感、排版节奏建立质感，而不是靠材质表演建立高级感。

### 4.2 必须做到
1. 先看懂，再好看。任务效率、扫描效率、状态识别优先于装饰。
2. 同一页面要有明确主工作面，不要把内容切成很多等权白卡。
3. 交互反馈要更像软件控件，不要出现漂浮、上抬、弹跳、发光感。
4. 页面必须适配 Desktop / Mobile，Light / Dark。
5. 对长中文、本地化、多状态和 reduced-motion 保持韧性。

### 4.3 明确禁止
1. 老式企业 OA：厚重、陈旧、机械、没有呼吸感。
2. 极客工具页：终端化、监控屏化、只对开发者友好。
3. AI 模板味：大白卡拼图、KPI 首页模板、双栏炫技登录、玻璃模糊、泛光渐变、按钮上抬。
4. 用“说明面板 + 卖点 bullet + 装饰背景”替代真实的信息层级设计。

### 4.4 可推翻与不可推翻
允许推翻：

1. 当前四页的版式。
2. 当前四页的视觉语言。
3. 当前四页的局部模块摆放方式。

不可推翻：

1. 页面承担的业务目标。
2. 关键字段、关键动作、关键状态。
3. 后端接口契约与能力边界。
4. 三层 edition 架构下的能力门槛。

## 5. 范围与边界
### 5.1 范围内
1. 管理员登录页 `/`
2. 服务总览 `/dashboard`
3. 账号池 `/accounts`
4. 批量导入 `/imports`
5. 以上页面的桌面端与移动端
6. 浅色与深色双主题
7. 关键反馈状态、空状态、加载状态、错误状态

### 5.2 范围外
1. 不改后端 API
2. 不改字段命名
3. 不重新定义业务流程
4. 不包含租户端门户的整套重设计
5. 不包含模型池、日志、计费、全局配置等其他页面

## 6. 技术与能力约束
### 6.1 API 与实现约束
1. 后端错误契约保持 `{ error: { code, message } }`
2. 不要求新增接口
3. 不要求新增复杂后端联动
4. 设计必须能映射到现有 React + Tailwind + shadcn 组件体系

### 6.2 Edition 约束
本产品同时支持 `personal / team / business` 三层架构，原型需要保留能力裁剪空间：

1. `personal` 不支持多租户与租户自助体系，因此不要把租户相关能力当成始终存在的硬前提。
2. `team` 支持多租户，但依赖更轻；设计不要默认强依赖复杂分布式能力。
3. `business` 支持完整能力；若某页存在“租户筛选 / API Key 筛选 / 更大体量数据”的扩展位，应在设计上允许扩展。

### 6.3 i18n 约束
1. 主文案可以先用中文设计。
2. 页面必须容纳更长的多语言文本，不能只对短英文成立。
3. 不要把技术 code 直接作为主可见文本。

## 7. 共享导航与信息关系
这四页都属于管理端工作台的一部分，应共享同一套成熟、克制的控制台语言。

页面关系如下：

1. 管理员登录页是后台入口。
2. 登录后默认进入 Dashboard。
3. Dashboard 是总览与分流页，应能导向日志、账单等深层页。
4. 账号池是资产管理页，与登录导入、批量导入形成连续资产操作链路。
5. 批量导入是工作台页，导入完成后结果会回流到账号池。

设计师需要理解：

1. Dashboard 负责“判断去哪里”。
2. 账号池负责“集中管理资产”。
3. 批量导入负责“把资产导进来并追踪任务”。
4. 登录页负责“快速、可信地进入整个系统”。

## 8. 页面级 PRD

## 8.1 管理员登录页
### 页面定位
这是管理员进入控制台的唯一入口页。它不承担品牌展示站职责，而承担“建立信任、明确身份、降低认知负担、快速进入系统”的职责。

### 目标
1. 在 5 秒内让用户理解这是“管理员受控登录入口”。
2. 明确主操作只有“登录”。
3. 让用户感受到系统安全、稳定、可追踪，而不是花哨。

### 核心内容
必须保留：

1. 页面标题与一句明确副标题。
2. 管理员用户名字段。
3. 管理员密码字段。
4. 主按钮“登录”。
5. 安全提示文案：连续登录失败会记录审计日志，方便后续排查。
6. 三条辅助价值点，语义围绕：
   - 审计可回溯
   - 管理链路稳定
   - 租户边界和凭据控制默认生效

### 功能与交互要求
1. 用户名默认预填 `admin`，但 UI 不必强化这一点。
2. 密码为空时，登录按钮不可用。
3. 登录中需要有明确“提交中”状态。
4. 登录失败必须有清楚、可理解的错误反馈。
5. 页面要支持语言切换、主题切换。

### 设计要求
1. 这是单一工作面，不要做模板化双栏炫技。
2. 品牌信息只能是边注或轻量支持，不应压过表单本身。
3. 登录页需要和管理后台整体风格一致，但更安静、更聚焦。
4. 移动端必须优先保证表单可达、主按钮清楚、提示不拥挤。

### 页面状态
必须覆盖：

1. 默认态
2. 输入中
3. 提交中
4. 登录失败
5. reduced-motion 兜底态

### 非目标
1. 不出现注册入口
2. 不出现租户端入口混用
3. 不做营销式产品介绍页

## 8.2 Dashboard（服务总览）
### 页面定位
这是管理员登录后的第一屏。它的职责不是“展示漂亮指标”，而是“帮助管理员快速判断当前系统哪里值得先看”。

### 核心任务
1. 快速读取系统运行脉搏
2. 切换查看范围
3. 查看关键趋势
4. 识别告警与异常
5. 继续跳转到请求日志或账单等深层页面

### 核心信息模块
必须保留以下结构语义，但允许设计师重组版式：

1. 页头信息
   - 页面标题：服务总览
   - 副标题：在一个总览视角里查看网关健康、用量变化与受管资源
   - 当前上下文信息：当前范围、当前时间窗、自动刷新说明
   - 快捷动作：查看请求日志、查看账单、刷新

2. 上下文与筛选
   - 范围筛选：全局 / 租户 / API 密钥
   - 时间范围：24 小时 / 7 天 / 30 天
   - 当范围不是全局时，需要租户筛选
   - 当范围是 API 密钥时，需要 API 密钥筛选

3. 运行脉搏
   用于快速读系统状态，至少要能承载以下语义：
   - 待处理告警数
   - 用量链路是否可用
   - 当前纳管范围
   - 可用上游库存

4. 指标读数
   至少包含以下 8 项：
   - 总请求数
   - Token 消耗总量
   - RPM
   - TPM
   - 平均首字速度
   - 租户数
   - 账号数
   - API 密钥数

5. Token 趋势
   - 按时间展示 Token 变化
   - 至少支持输入 / 缓存 / 输出 / 推理四个组件
   - 组件可单独开关

6. 模型分布
   - 展示 Top 模型
   - 支持按请求数 / 按 Token 两种模式切换

7. 待办告警
   - 需要支持搜索
   - 至少包含：级别、来源、告警内容、状态、时间、建议操作
   - 无数据时要有“系统运行稳定”类空状态

8. 热门 API 密钥
   - 展示排行榜
   - 需要与当前筛选范围联动

### 设计要求
1. 不允许做成典型 SaaS KPI 首页。
2. 指标必须更像“仪表读数”，不是“营销数卡”。
3. 页头、筛选、概览、趋势、告警之间要有明确阅读顺序。
4. 页面需要支持高频刷新和状态切换，不应显得戏剧化。
5. 右侧窄侧注区或依附式上下文区域是允许的，但必须服务于判断，不是装饰。

### 页面状态
必须覆盖：

1. 默认态
2. 加载态
3. 无趋势数据态
4. 有告警态
5. 无告警态
6. 刷新中态
7. 范围切换后态

### 设计重点
1. 第一眼就能知道“先看哪里”。
2. 告警和趋势要帮助决策，而不是增加视觉噪音。
3. 保持专业控制台气质，不要有首页展示页的表演感。

## 8.3 账号池
### 页面定位
这是管理员高频使用的核心运营页。它本质上是“账号资产管理台”，不是简单表格页。

### 核心任务
1. 快速查看当前账号是否可用
2. 批量筛选与批量操作
3. 查看账号细节
4. 触发刷新登录、启停、删除、同组暂停/恢复等动作
5. 从导入链路回到资产管理链路

### 核心信息模块
1. 页头信息
   - 标题：账号池
   - 副标题：在这里查看账号是否可用，并管理登录状态
   - 元信息：匹配数量、已选数量
   - 主动作：添加账号、导出、刷新

2. 筛选与批量操作区
   至少包含：
   - 状态筛选：全部 / 正常 / 已禁用
   - 账号类型筛选：全部 / OAuth 会话 / API 密钥
   - 凭据类型筛选：全部 / RT / AT / 未知
   - 套餐筛选
   - 匹配结果数
   - 已选数
   - 批量操作入口

3. 主表格
   至少容纳以下信息列：
   - 选择项
   - 账号身份（邮箱或标签）
   - 账号类型
   - 登录状态
   - 凭据类型
   - 套餐
   - 健康状态
   - Rate Limit 使用情况
   - 创建时间
   - 行级操作

4. 行级操作
   需要预留以下动作能力：
   - 查看详情
   - 刷新登录
   - 启用 / 禁用
   - 暂停同组账号
   - 恢复同组账号
   - 删除账号

5. 详情查看
   当前实现为弹窗式详情，设计师可以重新组织，但必须保留四类信息：
   - Profile：身份、连接信息、凭据、可用模型
   - OAuth：刷新状态、过期时间、来源类型、错误码、错误信息
   - Limits：额度与刷新时间
   - Raw：原始数据

### 设计要求
1. 这是高密度操作页，不应被设计成轻飘的展示页。
2. 筛选、批量操作、表格之间要像一个连续工作台，而不是三块不相干的卡片。
3. 详情查看必须能承载大量结构化信息，且不显杂乱。
4. 批量操作需要“可信、克制、有风险感知”，不能像普通轻操作。
5. 表格工具栏要体现专业软件的秩序，而不是表格组件默认皮肤。

### 页面状态
必须覆盖：

1. 默认态
2. 数据加载态
3. 刷新中态
4. 搜索 / 筛选后态
5. 选中多条后的批量操作态
6. 无匹配数据态
7. 查看详情态

### 设计重点
1. 让用户在高密度数据中仍然能迅速定位问题账号。
2. 强化“表格是主工作面”的感觉。
3. 让批量动作与风险动作有清晰层级，而不是全部同权。

## 8.4 批量导入
### 页面定位
这是典型工作台页面。它不是一个“上传页”，而是“导入前预检查 + 导入后追踪 + 导入结果回流”的完整任务面。

### 核心任务
1. 选择导入凭证模式
2. 批量加入文件
3. 在开始导入前完成预检查
4. 启动导入任务
5. 追踪任务进度
6. 查看最近任务与错误信息

### 核心信息模块
1. 页头信息
   - 标题：批量导入任务
   - 副标题：先批量加入文件，预检查后再一键导入，并实时查看每个任务的进度和错误明细
   - 元信息：支持上传格式、当前文件数、总大小

2. 导入模式选择
   必须支持：
   - 导入 RT
   - 导入 AK
   同时说明两者适用场景：
   - RT 适合托管续签和轮转
   - AK 适合一次性导入

3. 文件加入区
   必须支持：
   - 拖拽上传
   - 点击选择文件
   - 下载模板
   - 多文件一次加入

4. 预检查反馈
   至少包含：
   - 可导入文件数
   - 需确认文件数
   - 阻塞文件数
   - 估算总记录数
   - refresh_token / access_token / chatgpt_account_id 等关键统计

5. 待导入文件明细
   至少包含：
   - 文件名
   - 大小
   - 预检查结果
   - 状态
   - 移除动作
   - 搜索能力

6. 主提交动作
   - 开始导入（显示可导入文件数）
   - 当没有可导入文件时，需要明确阻塞反馈

7. 任务进度区
   至少包含：
   - 当前选中任务 ID
   - 任务状态
   - 导入进度百分比
   - 已入池 / 总数
   - 暂停跟踪 / 恢复跟踪
   - 取消任务

8. 最近任务队列
   - 展示最近导入任务
   - 可切换查看不同任务
   - 每项至少显示 job_id、处理进度、错误或状态摘要

### 设计要求
1. 这页的主角是“任务工作流”，不是上传控件本身。
2. 预检查反馈必须一眼可读，尤其要区分：
   - 可以直接导入
   - 需要人工确认
   - 当前阻塞
3. 右侧或次级区域适合放任务追踪，但不能与主上传工作面抢权。
4. 导入成功与失败都需要有稳定、专业的反馈，而不是轻飘 toast 风。
5. 页面要有“可安心操作”的感觉，因为这里直接影响账号资产。

### 页面状态
必须覆盖：

1. 默认空态
2. 拖拽悬停态
3. 本地预检查中
4. 存在阻塞文件态
5. 可以开始导入态
6. 导入任务进行中
7. 导入任务完成态
8. 导入任务失败态
9. 未选择任何任务时的右侧空态

### 设计重点
1. 把“预检查”做成真正可理解的决策辅助，而不是上传后的一堆技术反馈。
2. 让主任务面与任务追踪面关系清楚。
3. 让用户知道下一步该干什么，不需要自己猜。

## 10. 动效与交互原则
### 10.1 动效原则
1. 动效服务于状态切换和层级提示，不服务于展示。
2. 反馈应短、利落、克制。
3. 不使用 bounce、漂浮、卡片上抬、夸张 spring。
4. reduced-motion 下应退化为淡入或直接切换。

### 10.2 推荐动效强度
1. 页面进入：短时淡入或轻微位移，偏软件反馈。
2. 筛选切换：快速响应，不做大面积重编排秀场。
3. 导入进度：允许有稳定进度反馈，但不要用高戏剧化动画。
4. 详情查看：可有明确层级切换，但不能像营销弹窗。

## 11. 验收标准
1. 四页都明显摆脱模板化后台观感。
2. 第一眼能分清主工作面、次级上下文、主动作、风险动作。
3. Dashboard 更像总览控制台，而不是 KPI 首页。
4. 账号池更像高频运营台，而不是普通数据表。
5. 批量导入更像完整任务工作面，而不是上传弹窗放大版。
6. 登录页足够克制、可信、聚焦，不再像默认后台登录模板。
7. 双端、双主题下都能保持秩序与可读性。
8. 原型可映射到现有技术与 API，不要求后端重构。
