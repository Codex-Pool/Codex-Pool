# Frontend Page Archetypes Design

## 背景

本轮前端审计暴露的核心问题不是单个组件失误，而是页面表达缺少稳定的 archetype 约束：

- `auth` 页面仍有明显模板化舞台感，依赖 `Threads`、玻璃卡、发光文本等惯性表达。
- `workspace` 页面以 `ImportJobs` 为代表，仍带有展示页式 hero 结构，影响任务优先级和移动端负担。
- 同一个产品同时承载 `admin portal` 与 `tenant portal`，但不同页面类型缺少统一而可复用的页面语言。

在当前仓库规则与现有工作台风格已经明确的前提下，本次改造不再只修两页，而是先抽取一组页面 archetype，再用多类页面做首批落地样板。

## 目标

1. 建立稳定的前端页面 archetype，覆盖 `auth / dashboard / workspace / detail / settings` 五类场景。
2. 让认证页保留明确品牌舞台感，但摆脱“常见 AI 模板壳”的视觉特征。
3. 让工作台页回到“任务优先、状态清晰、信息克制”的基础设施产品表达。
4. 为后续 dashboard、settings、detail 页面收口提供一套可复用的布局与语义组件。

## 非目标

- 本轮不重做 dashboard 的数据结构与业务逻辑。
- 本轮不引入新的第三方设计系统或大规模替换现有基础组件。
- 本轮不修改后端接口或页面路由。

## 设计原则

### 1. 品牌表达集中，而不是平均铺满

品牌感应该集中在认证入口和少数概览节点，通过排版、比例、材质与节奏体现，而不是把每个页面都做成带强背景效果的展示页。

### 2. 页面先归类，再设计

同一产品内不同页面的视觉强度必须服从页面类型：

- `auth`：允许中高强度品牌叙事
- `dashboard`：中等强度，强调概览与优先级
- `workspace`：低到中等强度，先任务后装饰
- `detail`：低强度，强调追踪与上下文
- `settings`：最低强度，强调稳定和可预测

### 3. 质感来自结构，而不是特效

高端精致的感觉应来自：

- 清晰的空间节奏
- 经过推敲的标题与说明层级
- 有边界感但不过分厚重的表面材质
- 轻度色彩偏移的中性色体系

而不是依赖：

- `Threads`
- `ShinyText`
- 大面积 blur / glow
- 营销式 hero 区块

### 4. 页面标题区必须承担语义，而不是只是装饰

页头要清楚告诉用户：

- 这里是什么页面
- 当前主要任务是什么
- 当前筛选 / 作用范围是什么
- 下一步该做什么

说明文字不重复标题，强调信息密度和可扫读性。

### 5. 移动端优先处理任务路径

移动端不是桌面的缩小版。对 `workspace` 类页面，首屏应优先展示主任务区域和核心 CTA，次级统计与说明必须延后或折叠。

## Archetype 定义

### Auth Archetype

用途：管理员登录、租户登录、注册、邮箱验证、忘记密码等入口页面。

结构：

- 页面级背景使用安静、偏材质化的层次，不使用动态特效作为主表达。
- 左侧或上方为品牌舞台区：产品定位、信任锚点、少量辅助文案。
- 右侧或下方为操作区：表单、错误提示、次级入口。
- 品牌舞台区与操作区在视觉上解耦，避免“整块大玻璃卡双栏模板”。

要求：

- 品牌舞台保留，但不喧宾夺主。
- 表单是首要交互对象，错误与成功状态必须清晰。
- 在移动端，品牌叙事应压缩，不占用过多首屏。

### Dashboard Archetype

用途：管理员仪表板、租户仪表板、概览类首页。

结构：

- 短页头：标题、范围说明、主要筛选器、主操作。
- 主指标区：重点指标优先，次级信息不要并列抢焦点。
- 次级模块区：图表、摘要、快捷入口。

要求：

- 品牌感只能作为背景层或轻度材质，不使用展示页 hero。
- 所有筛选和刷新操作必须靠近上下文，不要漂浮成装饰元素。

### Workspace Archetype

用途：导入、上传、配置、排障、批处理、路由编辑等任务型页面。

结构：

- 短页头：标题、任务描述、必要说明。
- 主任务区：上传框、表单、主 CTA、关键状态反馈。
- 次级信息区：摘要卡、近期记录、日志、扩展详情。

要求：

- 主任务区必须在视觉上成为第一焦点。
- 预检统计、元信息、说明文本默认收敛为摘要，不抢主任务。
- 移动端优先保留主任务与反馈，把细节延后。

### Detail Archetype

用途：日志详情、请求详情、实体详情、审计追踪。

结构：

- 标识头部
- 关键状态
- 分组信息块
- 深入细节

要求：

- 强调可追踪性和对比阅读。
- 禁止额外舞台化。

### Settings Archetype

用途：系统设置、租户设置、计费配置等。

结构：

- 标题与说明
- 分段配置块
- 保存 / 重置操作

要求：

- 这是视觉上最克制的一类页面。
- 稳定性和可预测性优先于品牌表达。

## 共享原子与语义层

本次不会上完整设计系统，但会先抽一层足够复用的页面语义组件：

- `PageIntro`：标题、描述、作用范围、页头动作
- `BrandStage`：认证或入口页的品牌舞台区
- `WorkspaceShell`：任务页的页面节奏容器
- `WorkspacePrimaryPanel`：主任务面板
- `WorkspaceSecondaryPanel`：次级状态/详情面板
- `SectionEyebrow` / `SectionMeta`：小型语义文本

同时约束一组共享风格规则：

- 中性色表面优先，强调色只做点缀
- 大圆角减少到更利落的企业级比例
- 阴影减少厚重感，改为低层级材质感
- 页面说明文字缩短，避免重复标题

## 首批落地页面

### 1. Admin / Tenant Auth

目标：

- 保留舞台感
- 摆脱模板化双栏玻璃卡
- 让表单成为真正的交互中心

改造方向：

- 去掉 `Threads` / `ShinyText` 作为主表达
- 弱化整页玻璃壳，改成背景层 + 分离式舞台/表单区
- 通过更有秩序的标题、锚点文案、材质边界建立“高端精致”

### 2. ImportJobs

目标：

- 从展示页收回到工作台
- 主任务路径更清楚
- 移动端首屏负担明显降低

改造方向：

- hero 改为短页头
- 上传工作台成为主任务锚点
- 预检统计从“堆满首屏”改成“摘要优先、细节后置”

### 3. Admin / Tenant Dashboard

目标：

- 让 admin 与 tenant 概览页回到同一页面语言
- 把筛选、脉搏摘要、KPI 与图表拉回稳定的信息节奏
- 避免 admin 偏“运营看板模板”，tenant 偏“组件堆叠页”

改造方向：

- 使用共享 `DashboardShell / DashboardMetricGrid / DashboardMetricCard / SectionHeader`
- 页头保持“标题 + 作用范围 + 主动作”，不再做展示页 hero
- KPI 放在 rail 之前的移动端主内容流里，优先于筛选面板出现
- 桌面端 intro panel 不再被右侧 rail 拉伸成大块留白

## 验证标准

- 新结构能明显区分 `auth` 与 `workspace` 的表达强度。
- `auth` 看起来像产品入口，而不是常见生成式模板。
- `ImportJobs` 首屏能在 3 秒内让用户判断“下一步就是上传/开始导入”。
- 移动端无横向溢出，核心 CTA 不被摘要信息淹没。
- 不引入新的 i18n 硬编码、暗黑模式倒退或明显可访问性退化。

## 本轮落地结果

- `auth` 已改为“品牌舞台 + 操作面板”分离结构，桌面和租户入口截图均验证通过。
- `ImportJobs` 已改为短页头 + 主任务区 + 次级状态区，桌面与移动端截图均验证通过。
- `admin / tenant dashboard` 已迁移到共享 `dashboard` archetype，页头、筛选、脉搏摘要、KPI 与图表统一到同一节奏语言。
- `DashboardShell` 已修复两个真实走查暴露的问题：
  - mobile 上 rail 不再压在 KPI 前面
  - desktop 上 intro panel 不再被 rail 拉伸出大块空白
- 当前尚未扩展到 `detail / settings` 页面，但共享 archetype 层已经足够支撑下一批迁移。
