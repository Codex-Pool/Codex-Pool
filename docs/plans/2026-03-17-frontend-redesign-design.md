# Frontend Redesign Design

**日期：** 2026-03-17

## 背景

当前前端已经完成了一轮重要的结构抽象：

- `frontend/src/lib/page-archetypes.ts` 已定义 `auth / dashboard / workspace / detail / settings` 五类页面 archetype。
- `frontend/src/components/layout/page-archetypes.tsx` 已提供 `PageIntro`、`BrandStage`、`PagePanel`、`WorkspaceShell`、`ReportShell` 等共享页面骨架。
- 多个核心 admin / tenant 页面已经迁移到统一骨架，页面节奏比此前稳定得多。

这意味着项目已经不再处于“组件到处散落、不能动整体风格”的阶段，而是进入了“结构层可复用，但视觉系统层仍偏通用模板”的阶段。

当前真正的短板不在页面有没有组件，而在于：

- 全局 token 仍偏默认企业模板味
- 基础控件仍主要停留在 shadcn 默认风格
- 微交互分布零散，缺少统一节奏
- 剩余少数复杂页面尚未吃到统一 archetype

因此，现在适合开始重设计，但必须采用“先补设计基线，再做页面 rollout”的方式。

## 目标

1. 在不推翻现有信息架构和 edition/capability 约束的前提下，建立一套更成熟的前端视觉系统。
2. 让产品呈现“高端精致、可信克制、专业清晰”的控制台气质，而不是老式 OA 或开发者极客面板。
3. 用统一的微交互和反馈语言，提升页面的细腻感、状态感和使用流畅度。
4. 让 admin / tenant / personal 三层产品共享同一设计语言，但仍保留页面类型差异。

## 非目标

- 不重写业务数据流、权限模型或 capability gating。
- 不引入新的大型 UI 框架替换现有 React + Tailwind + shadcn 基础。
- 不把高频工作台页面做成营销站式 hero 或过度舞台化界面。
- 不在首轮改造中追求所有页面一次性换完。

## 方案对比

### 方案 A：渐进式系统重设计（采用）

做法：

- 先建立新的设计基线：颜色、排版、材质、阴影、圆角、动效、状态反馈。
- 再重做共享 primitive：按钮、输入、表格、页面表面、导航壳层。
- 最后按 archetype 分批迁移页面。

优点：

- 与现有 page archetype 抽象天然兼容
- 风险最可控，不容易把高频业务页做散
- 可随时验证每一批页面是否真的更好用

缺点：

- 需要一段时间同时维护“旧视觉 + 新视觉”的过渡状态

### 方案 B：先做一整套高保真新皮肤，再统一替换

做法：

- 先产出全局新视觉，再一次性批量覆盖页面

优点：

- 视觉上切换感强

缺点：

- 脱离真实页面约束，容易产出不耐用的“概念稿式设计”
- 页面细节、长文本、多语言和移动端韧性容易在落地时失真

### 方案 C：逐页自由重画

做法：

- 看见哪页旧就重画哪页，不先做基线

优点：

- 局部见效快

缺点：

- 最容易失去一致性
- 页面之间会快速漂移
- 微交互和控件语言很难统一

## 结论

采用方案 A。当前项目已经具备启动系统性重设计的结构条件，但还不具备跳过基线、直接逐页自由重做的条件。

## 设计北极星

新的前端视觉应更接近：

- 精密仪器
- 高端工业品
- 经过精修的企业控制台

而不是：

- 老式厚重企业 OA
- 赛博/霓虹/极客实验工具
- 玻璃拟态和大面积光效拼接的 AI 模板

关键词：

- 安静
- 准确
- 细致
- 有秩序的艺术感
- 含蓄但高级

## 视觉方向

### 1. 色彩

- 以石墨、云灰、矿物白、深墨作为主中性色。
- 强调色不再依赖默认蓝，而采用更克制的矿物青绿或冷金属蓝灰体系。
- 强调色只用于焦点、状态、选中和关键数据，不铺满页面。

### 2. 排版

- 放弃默认模板感较强的“到处 Inter + 一套普通字号”策略。
- 建立更有层次的标题、页头、说明、标签和数据字重体系。
- 标题要更像“编辑过的标题”，正文要更像“可靠的操作说明”。

### 3. 材质与表面

- 主表面使用轻微色温偏移和低强度边界，形成细腻层次。
- 阴影从“卡片掉在页面上”改为“材质轻轻浮起”。
- 圆角整体收敛，比当前更利落，但保留现代感。

### 4. 背景

- 工作区背景不再依赖明显装饰，而是使用更轻的纹理、雾化和光线秩序。
- 品牌感集中在登录、入口和概览区，工作页只保留非常克制的底层氛围。

## 微交互原则

### 1. 动效只服务状态变化

- 页面进入
- 面板展开/收起
- 表格与筛选切换
- 成功/失败反馈
- 导航展开/折叠

不做无意义悬浮、过度跟手和装饰性 bounce。

### 2. 节奏统一

- 页面级进入：短促、稳定、层次清楚
- 控件 hover：轻微提亮/提起，而不是大位移动画
- 焦点状态：清楚但不刺眼
- 加载状态：低噪声、不中断上下文

### 3. Reduced Motion 完整支持

- 所有微交互都必须有 reduced-motion 退化路径
- 低动效下仍要保留状态可感知性

## 页面分型落地策略

### Auth

- 允许最强品牌表达
- 品牌舞台与表单区保持分离
- 质感靠排版、材质、节奏，不靠炫光与特效

### Dashboard

- 页头收短
- KPI 和关键趋势优先
- rail 只做补充上下文，不抢主焦点

### Workspace

- 主任务区是第一视觉锚点
- 筛选、提示、摘要全部围绕任务组织
- 移动端必须优先保证主任务 CTA

### Detail / Settings

- 最安静、最稳定、最易扫读
- 通过信息层级和间距建立高级感，而不是装饰

## 第一阶段范围

第一阶段只做“设计基线”，不追求全面换肤：

1. 全局 design tokens 与 CSS 变量
2. 核心 primitive 的视觉升级
3. 导航壳层与登录入口的统一微交互
4. 选定少量样板页做视觉基线验证

首批样板页：

- `Login`
- `Dashboard`
- `ImportJobs`
- `Models`
- `Config`

## 风险与控制

### 风险 1：过度追求艺术感，削弱任务效率

控制：

- workspace / settings 页面一律先看任务路径，再谈装饰

### 风险 2：只改视觉，不改页面节奏

控制：

- 所有页面改造必须复用现有 archetype 和共享 shell

### 风险 3：视觉系统只覆盖 admin，不覆盖 tenant

控制：

- admin 与 tenant 的同类页面同步校对，避免双轨漂移

### 风险 4：微交互碎片化

控制：

- 先抽统一 motion preset，再允许页面使用

## 验证标准

- 即使不看 logo，也能感受到这是同一产品而不是拼装页面。
- 登录、工作台、设置、概览四类页面有明显分型差异，但气质一致。
- 交互表面、按钮、输入、表格、导航都拥有统一材质和细节语言。
- 微交互存在感明确但不过度喧宾夺主。
- 所有改造对多语言、长文本、浅色/深色、移动端和 reduced-motion 保持韧性。

## 结论

项目现在已经足够适合启动前端重设计，但正确起点不是“直接重画页面”，而是“先完成一层能承接长期 rollout 的设计基线”。基线一旦稳定，后续页面升级就会更快、更一致，也更不容易返工。
